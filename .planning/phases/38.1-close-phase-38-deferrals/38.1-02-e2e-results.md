## Script 08 failure — SQL Msg 8623

**Root cause:** Plan 02's `08-null-orphan-page-link-refs.sql` (commit `009ca20`, 214 lines) builds a **single monolithic UNION ALL pre-count query** by dynamic-SQL concatenation over `INFORMATION_SCHEMA.COLUMNS`. Against the fresh bacpac's ItemType_Swift-v2_* surface the query expands to:

- ~130 `ItemType_Swift-v2_*` BASE TABLE rows
- ~200 string columns (data_type `nvarchar` / `varchar` / etc.)
- Each string column gets **21 OR-branches** (20 × `Default.aspx?%=<id>` LIKE pair + 1 `TRY_CAST(... ) IN (idList)`)
- Plus ~50+ nullable integer columns × one `IN (idList)` branch

Total: a single `sp_executesql` string with ~8 000 LIKE predicates in one UNION ALL. SQL Server's query optimizer cannot produce a plan and throws Msg 8623 before mutation. `SET XACT_ABORT ON` triggers automatic ROLLBACK — the DB is unchanged by script 08, which is the safe outcome, but the 47-occurrence gap is NOT closed.

## Additional analysis — script 08 predicate also mis-derived

During triage a per-column rewrite of script 08 was prototyped and test-run against the live post-07 Swift-2.2 DB. With the 8623 compile failure eliminated, the rewritten pre-count reported **1641 rows** (far above the script's `[1..200]` upper-bound assertion), all in columns like `[Sort]`, `[Height]`, `[Id]`, `[FontSize]`, `[LineHeight]`, `[Padding]`, `[BorderThicknes]`, `[ItemSpacing]`, `[InitialZoomLevel]`, `[GroupNavigationStartLevel]`, etc. These are **legitimate integer-valued config columns** (font sizes = 19 px, padding = 2, sort orders = 40, etc.) — not page-link references. Part B (`TRY_CAST(...) IN (idList)`) and Part C (integer `IN (idList)`) false-positive on every integer-valued column whose value happens to equal one of the 20 common orphan IDs (1, 2, 4, 16, 19, 21, 23, 33, 34, 37, 40, 41, 42, 44, 48, 60, 97, 98, 104, 113).

Cross-check against the live bacpac (post-01-07 state) found:

| Scan | Matches |
|------|--------:|
| Form A only (`Default.aspx?ID=<orphan>` in any ItemType_Swift-v2_* string column) | **0** |
| Form A in BAK tables (pre-01 cleanup state) | **0** |
| Form A scoped to auto-discovered link-holding columns (columns currently containing any `Default.aspx?%ID=` string) | **0** |
| Form B + Form C (Plan 02 predicate) against all ItemType_Swift-v2_* | 1641 (but false-positives on config integers, not orphans) |

The 38.1-01 run's **47 `Unresolvable page ID` warnings surfaced on Deserialize Deploy** — NOT on Serialize Deploy. The warnings fire from `InternalLinkResolver.ResolveLinks` line 118 when the regex `Default\.aspx\?ID=(\d+)` matches a string that DW's `Item.SerializeTo(fields)` expanded from a link-typed integer column. `BaselineLinkSweeper.CheckField` (line 163-166) explicitly does NOT sweep raw-numeric references: *"raw-numeric references (plain `121` strings that should resolve to a page ID) are NOT swept here — too many false positives on ordinary numeric fields (sort orders, widths, etc.). The deserialize-time InternalLinkResolver still handles them via its 'entire string is a pure number AND is in the source-to-target map' check."*

**Implication:** The 47 orphan IDs are stored as **raw integers in specifically link-typed ItemType columns** (e.g., `CustomerCenterApp.OrdersPage = 98`). A SQL cleanup script cannot reliably identify these columns without DW's ItemType metadata (XML) — the `wwwroot/Files/System/Items/` XML folder is **incomplete** (no `ItemType_Swift-v2_CustomerCenterApp.xml` despite the DB table having 10+ `*PageLink` / `*Page` columns), and `ItemTypeDefinitions` in Swift-2.2 is empty (DW loads from XML at runtime). The 71 link-typed (table, column) pairs discovered from the XML folder are therefore incomplete and scoped scans against them return 0 matches.
