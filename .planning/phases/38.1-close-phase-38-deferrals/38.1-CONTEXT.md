## Implementation Decisions

### Scope shape

- **D-38.1-01:** Single plan with 6 sequential tasks (B.5.1 → B.4.1 → B.3.1 → GRID-01 → W6 → full live E2E). Smaller scope than Phase 38, no wave parallelization required. Each task commits atomically per the GSD executor pattern.

### B.5.1 — ButtonEditor SelectedValue paragraph-ID case

- **D-38.1-02:** Unconditional dual-check against `validIds` AND `validParagraphIds` in the `SelectedValuePattern` loop at `src/DynamicWeb.Serializer/Infrastructure/BaselineLinkSweeper.cs:167`. No JSON parsing of `LinkType` hints. Page IDs and paragraph IDs share no overlap in DW schema (separate identity seeds, separate tables); false-positive risk is vanishingly small. ~10 LOC + regression test.
- **D-38.1-03:** Regression test fixture must model the exact Swift 2.2 shape: a `ButtonEditor` JSON string containing `"SelectedValue": "15717"` with `"LinkType": "paragraph"` where 15717 is a paragraph ID NOT in the page tree. Test asserts the sweeper resolves (does NOT add to unresolved) when 15717 IS in `validParagraphIds`, and DOES add to unresolved when 15717 is absent from both sets.
- **D-38.1-04:** Remove the now-obsolete comment at BaselineLinkSweeper.cs:165-166 ("UNCHANGED. Per Pitfall 3, do NOT extend this pattern..."). Replace with a one-line comment documenting dual-check intent.

### B.4.1 — SHOP19 FK orphan cleanup

- **D-38.1-05:** Ship `tools/swift22-cleanup/06-delete-orphan-ecomshopgrouprelation.sql` deleting the orphan row `(ShopGroupShopId='SHOP19', ShopGroupGroupId='GROUP253')` from Swift 2.2's `[EcomShopGroupRelation]`. Follow the existing numbered-script shape (BEGIN TRANSACTION, count before, DELETE, count after, COMMIT). Update `tools/swift22-cleanup/README.md` to mark row 06 no longer `*(pending)*`.
- **D-38.1-06:** After the 06 script runs, the next re-serialize must produce a baseline YAML that no longer contains `baselines/Swift2.2/_sql/EcomShopGroupRelation/GROUP253$$SHOP19.yml`. Task's how-to-verify includes this assertion.

### B.3.1 — 7-column schema drift

- **D-38.1-07:** Doc-only resolution (B.3 Outcome A precedent). Extend `docs/baselines/env-bucket.md` §"DW NuGet version alignment" with the 7 newly-observed columns grouped by table: `EcomGroups.GroupPageIDRel`; `EcomProducts.ProductPeriodId`, `ProductVariantGroupCounter`, `ProductPriceMatrixPeriod`, `ProductOptimizedFor`, `MyVolume`, `MyDouble`. Same guidance as the 3 Area columns: align DW NuGet versions between source and target hosts.
- **D-38.1-08:** No new config field (`knownEnvSchemaDrift` explicitly rejected — opens config surface to warning-silencing abuse). No curated static map in library (doesn't generalize beyond Swift 2.2). No StrictModeEscalator warning-classification change (deferred — see Deferred Ideas). If a customer hits this and cannot align DW versions, the escalation path is a future phase, not this one.
- **D-38.1-09:** `TargetSchemaCache.LogMissingColumnOnce` stays unchanged. The warning it emits is correct and intentional. Strict mode escalation is also correct — it surfaces a real source/target mismatch that the customer should fix operationally.

### GRID-01 — 142 stale-email GridRows

- **D-38.1-10:** Source SQL cleanup, not serializer-side coalesce. Ship `tools/swift22-cleanup/07-delete-stale-email-gridrows.sql` deleting the 142 GridRows with `GridRowDefinitionId IN ('', '1ColumnEmail', '2ColumnsEmail')` from Swift 2.2. Matches B.1/B.2 upstream-cleanup pattern (Phase 38-03). No code change in `ContentDeserializer` or `GridRow` write path — serializer should NOT learn to accept NOT NULL violations silently.
- **D-38.1-11:** Expected row count 142 (asserted in the script's pre-delete SELECT). If the count differs at execution time, the script aborts (no DELETE runs). The post-delete SELECT must return 0.
- **D-38.1-12:** Update `tools/swift22-cleanup/README.md` to add row 07 describing the delete + count.
- **D-38.1-13:** Execution order dependency — the 07 script requires that Phase 38-03's 05-null-stale-template-refs.sql has already run (which sets the 142 rows' `GridRowDefinitionId` to `''`). README must document this ordering.

### W6 — shared paragraph-ID collector helper

- **D-38.1-14:** Extract `BaselineLinkSweeper.CollectSourceParagraphIds` and the parallel logic in `src/DynamicWeb.Serializer/Serialization/InternalLinkResolver.cs` into a shared static helper. Target location: `src/DynamicWeb.Serializer/Infrastructure/ParagraphIdCollector.cs` (Infrastructure to match BaselineLinkSweeper's namespace). Both call sites update to use the helper. Risk: InternalLinkResolver has its own invariants; extraction must not regress existing tests. Plan's how-to-verify runs both classes' existing test suites.

### Bucket rename — ENVIRONMENT → NOT-SERIALIZED

- **D-38.1-16:** Doc-rename only (no code mentions the term). Replace `ENVIRONMENT` with `NOT-SERIALIZED` in:
  - `docs/baselines/Swift2.2-baseline.md` §"The three-bucket split" + all downstream references

- **D-38.1-18:** Keep the 3-bucket structure. No evidence in `.planning/` or docs of a prior decision to collapse to 2 buckets. User-raised possibility closed with clarification that ENVIRONMENT is not "env-specific YAML" — it's "never in YAML at all".

## Specific Ideas

### The B.5.1 test fixture exact shape

From `38-05-e2e-results.md` live observation, the Swift 2.2 source has:
```json
{ "SelectedValue": "15717",
  "Label": "Become partner",
  "Link": "Default.aspx?ID=4897#15717",
  "LinkType": "paragraph",
  "Style": "outline-primary" }
```
in `[ItemType_Swift-v2_Poster].[SecondButton]`. Test fixture should model a `SerializedPage` → `Paragraph` → `Fields["SecondButton"]` with this JSON string verbatim (or minimal structurally-equivalent form).

### The GRID-01 doomed-row count

From `38-03-SUMMARY.md`: exactly 142 rows in `[GridRow]` where `GridRowDefinitionId IN ('', '1ColumnEmail', '2ColumnsEmail')` after Phase 38-03's 05 script nulled the template names to empty string. The 07 script must assert this count before DELETE.

### The NOT-SERIALIZED bucket definition clarification

Canonical phrasing to add to `Swift2.2-baseline.md §The three-bucket split`:

> **NOT-SERIALIZED** (formerly "ENVIRONMENT"): data that lives outside any baseline YAML because it varies per environment and is owned by the target host's operator. If it can appear in a YAML file, it is by definition DEPLOYMENT or SEED, not NOT-SERIALIZED.

```csharp
internal static class ParagraphIdCollector
{
    public static void CollectSourceParagraphIds(
        IEnumerable<SerializedPage> pages,
        HashSet<int> sink);

```

Both call sites adopt it; existing tests for both classes must stay green.

<deferred>
## Deferred Ideas

- **StrictModeEscalator warning classification (option D for B.3.1)** — treating "source column not present on target schema" as a non-fatal warning class while keeping other warnings escalating. Rejected for 38.1 as scope creep. Revisit if multiple customers hit schema-drift and can't align DW versions; that signals the operational fix path isn't sufficient.
- **`knownEnvSchemaDrift` config field (option B for B.3.1)** — per-predicate allowlist. Rejected for 38.1 to avoid opening config surface to warning-silencing abuse. Revisit only if options A + D both prove inadequate.
- **Curated static runtime map for drift (option C for B.3.1)** — internal Dictionary of known-drift columns. Rejected as non-generalizable. Only tenable if the drift list stays small and stable.
- **Collapse 3-bucket model to 2 buckets** — user asked whether this was prior work. It wasn't. Rejected as structural change outside 38.1 scope. File a separate ideation phase if it ever becomes important.
- **Serializer-side GridRow empty-DefinitionId coalesce** — rejected for GRID-01 in favor of upstream SQL cleanup. If a future customer's baseline has legitimate empty-DefinitionId rows that should survive, reconsider.
- **Parse `LinkType: paragraph` JSON hint in B.5.1** — rejected in favor of unconditional dual-check. If false positives ever appear (page+paragraph ID collision — theoretically impossible in DW schema), add JSON-aware gating then.
- **CI smoke integration for the live E2E recipe** — ROADMAP already defers tools/smoke CI integration per D-38-13. 38.1 runs the E2E locally by hand; automation is a future phase.
