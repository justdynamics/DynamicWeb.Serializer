<failure_handling>
## If the pipeline fails

The pipeline writes detailed logs and throws on every gate failure. Fallback protocol:

1. **Read the failing step's log** from the run directory.
2. **If the failure is a bacpac restore issue (sqlpackage install, DB drop fail, bacpac corrupt):** stop and report to orchestrator — this is NOT a serializer bug, it is an infrastructure issue.
3. **If the failure is a cleanup-script ABORT (pre-count mismatch):** read the script's expected count vs observed count from the log. This means the bacpac baseline differs from the 38.1-01-e2e-results observation (e.g. the bacpac may have been captured before scripts 01-07 applied). If the script aborted cleanly (ROLLBACK, no DB change), record the mismatch in a halt report and hand back to the orchestrator — scripts 08/09 predicates may need re-derivation.
4. **If the failure is a serialize HTTP non-200:** read `serialize-deploy.log` or `serialize-seed.log`. The most likely cause is a BaselineLinkSweeper error for a source-data shape not covered by scripts 01..08 — report the escalation list.
5. **If the failure is a deserialize HTTP non-200:** read `deserialize-deploy.log`. The escalation list names every warning that escalated; compare against the 20 orphan IDs + PageID 88/103 set. Any escalation OUTSIDE this set is a new gap (out-of-scope for this plan).
6. **If the smoke tool runs but reports "Nothing to test":** CleanDB did not receive pages. Check `deserialize-deploy.log` for partial-failure — the deserialize may have returned 200 but written 0 pages due to a data-layer issue.

In all failure paths, halt the plan. Do NOT attempt to fix in-flight. Return a structured failure report to the orchestrator with the exact log reference.
</failure_handling>
