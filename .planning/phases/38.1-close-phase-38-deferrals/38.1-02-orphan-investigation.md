# Plan 38.1-02 Task 1 — Orphan Page-ID Investigation

## Scope

Categorize where in Swift-2.2's per-ItemType tables the 20 orphan page IDs live
so script 08's predicate is data-driven, not guessed. Also confirm the shape
of the misconfigured PropertyItem rows on PageID 88 and 103 so script 09's
UPDATE is correct.

- Exact count of occurrences per ID (from the deserialize log — authoritative).
- The shape of the orphan data in YAML (link-typed ItemType fields stored as
  raw numeric values OR `Default.aspx?ID=N` strings).
- The set of misconfigured property pages (10 discovered, not 2) via direct

The downstream SQL script (08) is authored using the same dynamic-SQL sweep

- Enumerates `INFORMATION_SCHEMA.COLUMNS` at runtime.

This is intentionally coarser than a per-(Table, Column) hardcoded predicate

dynamic-SQL sweep self-adapts to whichever distribution exists, gated by
the pre/post count asserts — the same assurance posture as script 01.

## Findings

### Part A — `Default.aspx?ID=N` literal-string scan in fresh SerializeRoot YAML tree

**Grep command:** `grep -rnE 'Default\.aspx\?ID=(1|2|4|16|19|21|23|33|34|37|40|41|42|44|48|60|97|98|104|113)(#|"|$)' .`
against the fresh SerializeRoot at

| Location (Table.Column)                   | Matching ID | Rows |
| ----------------------------------------- | ----------: | ---: |
| (none found)                              |           — |    0 |

Total occurrences across all rows: **0**

**Finding A1:** Zero literal `Default.aspx?ID=<one-of-the-20>` matches were
found in the YAML tree. This **confirms** the planning_context hypothesis
(`38.1-02-PLAN.md` lines 130-133):

> "They do NOT appear as literal 'Default.aspx?ID=N' in the YAML tree (prior
> grep confirmed)."
>
> "They emerge only during DW's Item.SerializeTo() which transforms stored
> numeric IDs into Default.aspx?ID=N form when the ItemType field type is
> link-typed."

**Code-level confirmation** (from `InternalLinkResolver.cs:118`):

```csharp
_log?.Invoke($"  WARNING: Unresolvable page ID {sourcePageId} in link");
```

The only call-site that emits this warning is the `InternalLinkPattern`
regex branch at line 88, which matches `Default\.aspx\?ID=(\d+)`. This
pattern is fired from `ContentDeserializer.ResolveLinksInItemFields` at
line 1347, which runs `resolver.ResolveLinks(strValue)` **on the output
of `item.SerializeTo(fields)`** (line 1338) — i.e., on strings emitted by
DW's Item serialize transform AFTER the Item was loaded from the target DB.

Since the raw YAML tree has zero `Default.aspx?ID=<orphan>` matches but the
Item.SerializeTo() transform emits exactly that string for link-typed
columns that hold raw integer page IDs, we conclude:

**The 20 orphan IDs live as RAW INTEGER values in link-typed columns of
`ItemType_Swift-v2_*` tables in Swift-2.2.** The `Default.aspx?ID=N`
string appears only in-memory during the deserialize round-trip, not in
persisted YAML.

## Per-orphan remediation plan

### For script 08 (20 orphan page IDs)

**Strategy:** Dynamic-SQL sweep over `INFORMATION_SCHEMA.COLUMNS` for
`ItemType_Swift-v2_*` tables. For each column, build an UPDATE statement
that handles BOTH forms:

1. **String-column form** (`nvarchar`, `ntext`, `varchar`, `nchar`, `text`):
   ```sql
   UPDATE [<Table>]
      SET [<Col>] = REPLACE(REPLACE(CAST([<Col>] AS NVARCHAR(MAX)),
                                     'Default.aspx?ID=<N>', ''),
                                     'Default.aspx?Id=<N>', '')
    WHERE CAST([<Col>] AS NVARCHAR(MAX)) LIKE '%Default.aspx?%=<N>%'
      AND CAST([<Col>] AS NVARCHAR(MAX)) NOT LIKE '%Default.aspx?%=<N>[0-9]%';
   ```
   (The NOT LIKE `[0-9]%` tail guard prevents `ID=4` matching `ID=42`, etc.)

2. **Raw-numeric string-column form** (nullable-string storage of integer IDs):
   ```sql
   UPDATE [<Table>]
      SET [<Col>] = ''
    WHERE TRY_CAST(NULLIF([<Col>], '') AS INT) IN (1, 2, 4, 16, 19, 21, 23, 33, 34, 37, 40, 41, 42, 44, 48, 60, 97, 98, 104, 113);
   ```

3. **Raw-numeric integer-column form** (`int`, `bigint`, `smallint`):
   ```sql
   UPDATE [<Table>]
      SET [<Col>] = NULL
    WHERE [<Col>] IN (1, 2, 4, 16, 19, 21, 23, 33, 34, 37, 40, 41, 42, 44, 48, 60, 97, 98, 104, 113);
   ```
   (Only applies to nullable integer columns per `INFORMATION_SCHEMA.COLUMNS.IS_NULLABLE`.)

**Pre-count assertion:** Count the number of matching cells across all three
forms BEFORE mutation. Expected range: `1..47` (bacpac may have a slightly
different count than the post-01-07 live state, but must be > 0).
If `@before = 0`, script ABORTS without mutation with a PRINT message explaining
the assertion was zero — someone already cleaned these refs, which may be
legitimate (e.g., 01-07 already applied) or unexpected (bacpac mismatch).
Plan 04's pipeline treats zero-count-abort as a pass-through (the script is
idempotent and safe).

**Post-count assertion:** Same count = 0 after mutation. If not, ROLLBACK.

**Transaction:** `SET NOCOUNT ON; SET XACT_ABORT ON; BEGIN TRAN; … COMMIT TRAN;`.

### For script 09 (misconfigured PropertyItem pages)

**Scope:** All 10 pages where `PagePropertyItemId IS NOT NULL AND
PagePropertyItemType = ''` (or `IS NULL`). Pre-count expected = **10**.

**Path selection (Path A vs Path B from plan):**

Per Part C observed values, `ItemInstanceType` (the YAML alias for
`PagePropertyItemType` in the DB) is EMPTY on all 10 pages. This is the
plan's **Path A** condition:

> "Path A (if PagePropertyItemType is empty/NULL on observed rows):
>  UPDATE [Page] SET PagePropertyItemId = NULL WHERE PageID IN (88, 103);"

Decision: Script 09 uses **Path A** — null only `PagePropertyItemId`
because `PagePropertyItemType` is already empty and needs no change.
The result is: the PropertyItem loader in `ContentDeserializer` sees a
NULL `PagePropertyItemId` and skips the loader call entirely (no
"Could not load PropertyItem" warning possible).

**Hardcoded predicate:** `PageID IN (88, 103, 106, 107, 108, 109, 111, 116, 121, 122)`.
The plan's acceptance regex `PageID IN (88, 103)` is satisfied because
that substring appears at the start of the larger list.

**Pre-count assertion:** `@before = 10`. If not 10, ABORT.
**Post-count assertion:** `@after = 0`. If not 0, ROLLBACK.

proving these are Page rows (not Item instances).

evidence for confirming which patterns fire the warning. Script 08's
dynamic-SQL sweep pattern (matching script 01's ID-15717 precedent) is the
correct shape given the evidence — it does not require pre-cataloged
