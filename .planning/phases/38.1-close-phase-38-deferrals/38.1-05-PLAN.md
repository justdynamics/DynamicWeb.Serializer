       The Swift 2.2 reference install carries a set of data-hygiene issues that
       never surface in normal admin operation but do surface when running a
       strict-mode baseline round-trip (Swift-2.2 → YAML → Swift-CleanDB). Six
       categories of cleanup surfaced across Phase 37, Phase 38, and Phase 38.1;

       reference baseline can be refreshed whenever upstream Swift ships new

       ## Cleanup categories

       ### 1. FK orphans

       **Location:** `[EcomShopGroupRelation]` table (`ShopGroupShopId`, `ShopGroupGroupId`).
       **Problem:** One row referenced a non-existent shop (ShopGroupShopId = `SHOP19` / ShopGroupGroupId = `GROUP253`) while Swift 2.2 only ships 9 shops (SHOP1 / 5 / 6 / 7 / 8 / 9 / 14 / 27 / 28). When CleanDB deserialize re-enables the EcomShopGroupRelation → EcomShops FK, the orphan row causes a strict-mode escalation.
       **Script:** `tools/swift22-cleanup/06-delete-orphan-ecomshopgrouprelation.sql`
       **Expected rows:** 1 before → 0 after.
       **E2E gate unblocked:** SqlTableProvider FK re-enable on deserialize.

       ### 2. Stale email GridRows

       **Location:** `[GridRow]` table (`GridRowDefinitionId` column).
       **Problem:** 142 GridRow rows point at email grid-row templates (`1ColumnEmail`, `2ColumnsEmail`) that upstream Swift no longer ships. After script 05 nulls the template names to `''`, the NOT NULL constraint still admits empty strings locally, but on deserialize into CleanDB the GridRow writer fails with 142 NOT NULL violations.
       **Scripts:**
       - `tools/swift22-cleanup/05-null-stale-template-refs.sql` (prerequisite — nulls 1ColumnEmail / 2ColumnsEmail / Swift-v2_PageNoLayout.cshtml references to `''`)
       - `tools/swift22-cleanup/07-delete-stale-email-gridrows.sql` (deletes the 142 empty-name rows)
       **Expected rows (script 07):** 142 before → 0 after.
       **E2E gate unblocked:** ContentDeserializer GridRow insert path.

       ### 3. Orphan link refs (pages that no longer exist)

       **Location:** `ItemType_Swift-v2_*` tables' link-typed columns across ~91 per-ItemType tables. Detailed per-(Table, Column, ID) breakdown in `.planning/phases/38.1-close-phase-38-deferrals/38.1-02-orphan-investigation.md`.
       **Problem:** Paragraph / ItemType field values reference page IDs that do not exist in the Page table. Two groups were surfaced across the round-trip history:
       - **Group A (Phase 37, 5 IDs):** 8308 (Area 26 cross-baseline), 149, 15717 (paragraph ID stored as `{"SelectedValue": "15717"}` in ButtonEditor JSON), 295, 140.
       - **Group B (Phase 38.1, 20 IDs):** 1, 2, 4, 16, 19, 21, 23, 33, 34, 37, 40, 41, 42, 44, 48, 60, 97, 98, 104, 113.
       Group A surfaces as literal `Default.aspx?ID=<N>` strings in Swift-v2 ItemType columns; Group B surfaces via DW's `Item.SerializeTo()` which transforms stored numeric link-field values into `Default.aspx?ID=<N>` form during serialize.
       **Scripts:**
       - `tools/swift22-cleanup/01-null-orphan-page-refs.sql` (5 earlier IDs)
       - `tools/swift22-cleanup/08-null-orphan-page-link-refs.sql` (20 later IDs; 47 occurrences)
       **E2E gate unblocked:** BaselineLinkSweeper serialize-time sweep + strict-mode Deserialize Deploy HTTP 200.

       ### 4. Misconfigured property pages

       **Location:** `[Page]` table (`PagePropertyItemId`, `PagePropertyItemType` columns).
       **Problem:** PageID 88 ("Navigation") and PageID 103 ("Secondary Navigation") both have `PagePropertyItemId` set to a value without a matching `PagePropertyItemType`. The PropertyItem loader on deserialize cannot resolve the dangling reference.
       **Script:** `tools/swift22-cleanup/09-fix-misconfigured-property-pages.sql`
       **Expected rows:** 2 before → 0 after.
       **E2E gate unblocked:** PropertyItem loader on deserialize.

       ### 5. Stale ItemType title references

       **Location:** `wwwroot/Files/System/Items/ItemType_Swift-v2_*.xml` (host filesystem, NOT the DB).
       **Problem:** 7 Swift-v2 ItemType definitions declare `fieldForTitle="Title"` but do not include a `Title` field in their field collection. Admin UI renders "Field for title" as a stale reference.
       **Status:** observed but not yet scripted. Detailed analysis in `docs/findings/swift22-stale-itemtype-title-references.md`.
       **E2E gate:** relates to 7 of the 47 Group-B page-ID warnings (see the stale-title finding doc §"Related strict-mode round-trip symptom") — cleanup is by XML edit, not SQL; a future phase may automate it.

       ### 6. Operational target-schema alignment

       **Location:** `[Area]`, `[EcomGroups]`, `[EcomProducts]` tables on CleanDB (the target).
       **Problem:** Swift 2.2 ships 10 columns that upstream DW NuGet 10.23.9 on CleanDB does not: `Area.AreaHtmlType`, `Area.AreaLayoutPhone`, `Area.AreaLayoutTablet`, `EcomGroups.GroupPageIDRel`, `EcomProducts.ProductPeriodId`, `EcomProducts.ProductVariantGroupCounter`, `EcomProducts.ProductPriceMatrixPeriod`, `EcomProducts.ProductOptimizedFor`, `EcomProducts.MyVolume`, `EcomProducts.MyDouble`. Without alignment, `TargetSchemaCache.LogMissingColumnOnce` emits 10 warnings per deserialize; strict mode escalates them all.
       **Script:** `tools/swift22-cleanup/cleandb-align-schema.sql` (idempotent `IF COL_LENGTH IS NULL` guards; runs on CleanDB, not Swift-2.2).
       **E2E gate unblocked:** zero schema-drift warnings on CleanDB deserialize.
       **Rationale source:** `docs/baselines/env-bucket.md` §"DW NuGet version alignment".
