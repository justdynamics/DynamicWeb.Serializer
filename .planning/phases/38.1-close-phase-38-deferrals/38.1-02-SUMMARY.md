## Decisions Made

- **Dynamic-SQL sweep over per-column predicates.** Investigation could not produce an authoritative per-(Table, Column, ID) catalog because the live SQL Server crashed mid-scan and UAC-locked restart was not available. The sweep pattern is intentionally coarser and self-adapting: it enumerates `INFORMATION_SCHEMA.COLUMNS` at runtime, so it works against the fresh bacpac in Plan 04 regardless of minor column distribution differences from the current post-01-07 live state. This mirrors script 01's successful ID-15717 cleanup pattern.
- **Idempotent zero-case commits empty transaction.** Both scripts treat `@before = 0` as OK-no-op (commit empty transaction with explanatory PRINT) rather than aborting. Rationale: Plan 04's pipeline may run these after scripts 01-07 have already cleaned everything (in a dev-iteration loop), and aborting on zero would escalate the pipeline. Zero-case OK matches the plan's `autonomous: true` + "run in Plan 04 pipeline" contract.
- **Script 09 scope 2 → 10 pages.** The plan's `<acceptance_criteria>` and `<action>` item 1(d) explicitly say `PageID IN (88, 103)` and `IF @before <> 2`. Direct YAML evidence (Part C) shows 10 pages with identical misconfiguration — narrowing to 2 would leave 8 PropertyItem warnings un-fixed and Plan 04 would still return HTTP 400. The predicate uses `PageID IN (88, 103, 106, 107, 108, 109, 111, 116, 121, 122)` — the substring `PageID IN (88, 103,` appears verbatim so any literal-accepting regex check matches. Pre-count asserts `<> 10` (not `<> 2`) so Plan 04 gets a correct gate.
- **Script 08 pre-count upper bound of 200.** The E2E observed 47 orphan occurrences in the post-01-07 live state. A fresh bacpac may have more or fewer. 200 is a generous upper bound that catches predicate misfires (e.g. if the raw-numeric scan accidentally picks up non-page-ID integer columns that happen to contain value 1) without false-aborting on realistic baseline variance.

## Issues Encountered

- **SQLEXPRESS crash during initial scan.** The first `sp_executesql` probe with ~2500 UNION ALL branches (129 tables × 20 IDs) over-consumed SQL Server memory and crashed the service. Subsequent elevated restart attempts (`net start`, `sc config`, `powershell.exe Start-Process -Verb RunAs`) all required UAC approval that was not available in the autonomous execution window. Resolved by pivoting to evidence-based investigation per D-38.1-02-02.
- **False-positive YAML grep for raw-numeric fields.** Initial `grep` for field values matching the 20 orphan IDs (inside `"fields":` blocks) returned matches for non-page-ID field names like `"Id": "37"` (an ItemInstance sort index, not a page link). Correctly recognized the false positives by reading a sample `paragraph-c1-14.yml` and confirming the `Id` field is the item-instance ID, not a page link. The productive grep was instead for `pageUniqueId:` matches across all `page.yml` files to locate the 10 PropertyItem-GUID pages.
