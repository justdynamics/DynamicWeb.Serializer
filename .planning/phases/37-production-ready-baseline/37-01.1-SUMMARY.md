## Decisions Made

1. **ConfigPath test hook on queries** — The four Item Type / XML Type queries grew a `public string? ConfigPath { get; set; }` test override paralleling the existing command pattern. Without it, query tests would have to either populate candidate paths on disk (fragile, clashes across test classes) or skip direct query coverage. The hook is `null` in production code paths.
2. **ContentSerializer / ContentDeserializer pinned to Deploy explicitly** — Neither class has a `DeploymentMode` parameter today; they're consumed via `ContentProvider`, which builds a one-predicate `SerializerConfiguration` with only `Predicates` populated. After removing the legacy alias, a choice is needed: (a) thread `DeploymentMode` through both classes and through `ContentProvider.BuildSerializerConfiguration`, or (b) pin to Deploy now and defer the thread-through. Chose (b) — behaviour-preserving (the alias also forwarded to Deploy), and the SEED-mode tree path is purely UI-editable at this layer; runtime Seed semantics in ContentSerializer remain a documented follow-up from 37-01.
3. **Per-mode leaf prefixes over shared prefix** — Every ItemType/XML leaf node under Deploy and Seed has a distinct ID (`Serializer_Deploy_ItemType_{SystemName}` vs `Serializer_Seed_ItemType_{SystemName}`). Alternative: shared leaf ID under a different parent chain. Rejected — DW's NavigationNodePath uses the full ID, not the parent chain, for highlighting; a shared leaf ID would highlight both Deploy and Seed branches on a single-leaf focus.
4. **NodeId constants promoted to public** — `SerializeNodeId`, `PredicatesNodeId`, `Deploy*`/`Seed*` NodeId constants changed from `internal const` to `public const`. They're referenced across the tree layer and now by tests; they're also user-visible in the DW admin URL. Public is the accurate access level.
5. **AsyncLocal for TestOverridePath** — A plain `public static string? TestOverridePath` leaked across parallel xUnit workers during the full-suite run, breaking the ConfigPathResolverTests race even when the tree test's `Dispose` restored the saved value. `AsyncLocal<string?>` scopes the override to the test worker's flow, preventing leakage.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] ConfigPath test hook added to queries**
- **Found during:** Task 1 RED
- **Issue:** Plan called for query-level tests (`ItemTypeBySystemNameQuery_WithDeployMode_ReadsFromDeployDict` etc.), but none of the four queries had a `ConfigPath` test-override property. Tests would have had to write the config to a disk-resolver candidate path, which is fragile under parallel xUnit.
- **Fix:** Added `public string? ConfigPath { get; set; }` to `ItemTypeListQuery`, `ItemTypeBySystemNameQuery`, `XmlTypeListQuery`, `XmlTypeByNameQuery` — mirroring the pattern already on `SaveItemTypeCommand` / `SaveXmlTypeCommand`. Production code paths leave it null and fall back to `ConfigPathResolver.FindConfigFile()`.
- **Files modified:** 4 query files (see key-files above).
- **Verification:** All Task 1 tests that rely on the hook pass.
- **Committed in:** `ea91d2d` (Task 1 GREEN).

**2. [Rule 1 - Bug] AsyncLocal scoping for TestOverridePath**
- **Found during:** Task 2 GREEN (full-suite test run)
- **Issue:** After adding `public static string? TestOverridePath` to `ConfigPathResolver`, a full-suite run exhibited intermittent `ConfigPathResolverTests.FindOrCreateConfigFile_*` failures. Root cause: my tree tests' `Dispose()` restores a saved override value, but the override leaked across parallel xUnit workers BETWEEN the constructor running and Dispose on the other fixture class — a cross-test-class data race.
- **Fix:** Backed `TestOverridePath` with `AsyncLocal<string?>` so each test's async flow gets its own isolated slot. Tree tests' constructor/Dispose still work via the property accessor, but the value is scoped to the flow that set it.
- **Files modified:** `src/DynamicWeb.Serializer/Configuration/ConfigPathResolver.cs`
- **Verification:** Two consecutive clean full-suite runs at 435/435 after the fix.
- **Committed in:** `41151b1` (Task 2 GREEN — folded in with the tree provider changes since it was found in the same test pass).

**3. [Rule 2 - Missing Critical] Mode field on ItemTypeListModel + XmlTypeListModel**
- **Found during:** Task 2 GREEN (implementing path provider)
- **Issue:** Plan didn't explicitly call out adding `Mode` to the list models, but `ItemTypeNavigationNodePathProvider` receives an `ItemTypeListModel` and must terminate at the correct per-mode `ItemTypesNodeId` (`DeployItemTypesNodeId` vs `SeedItemTypesNodeId`). Without `Mode` on the list model, the provider has no way to pick the right terminal.
- **Fix:** Added `DeploymentMode Mode` to both list models and populated it from the query's `Mode` when materialising rows. The path provider now reads `model?.Mode ?? DeploymentMode.Deploy` and picks the right terminal.
- **Files modified:** `src/DynamicWeb.Serializer/AdminUI/Models/ItemTypeListModel.cs`, `XmlTypeListModel.cs`, `ItemTypeListQuery.cs`, `XmlTypeListQuery.cs`, `ItemTypeNavigationNodePathProvider.cs`.
- **Verification:** Tree tests pass; tree-highlight correctness is indirectly tested via `Root_ContainsTwoModeNodes_EachWithItemTypesAndXmlTypesChildren` + `NodeIds_AreUniqueAcrossModes`.
- **Committed in:** `41151b1` (Task 2 GREEN).
