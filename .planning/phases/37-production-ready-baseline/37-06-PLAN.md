<objective>
Close the single blocking gap in Phase 37 verification: **SC-3 (SQL identifier validation at config-load) fails in production paths because every production call site uses the 1-arg `ConfigLoader.Load(path)` overload, which passes `identifierValidator: null` and skips `ValidateIdentifiers`.**

The validator infrastructure (`SqlIdentifierValidator`, `SqlWhereClauseValidator`, `ValidateIdentifiers` helper) is fully implemented and tested. Only the wire-through is missing. This plan flips the default: the 1-arg overload now constructs a default `SqlIdentifierValidator()` and delegates to the 2-arg overload, closing SC-3 for every production call site (22 locations) without touching any of them.

**Same finding as 37-REVIEW.md CR-01** â€” both reports independently converged on the same root cause. 37-03-SUMMARY.md explicitly flagged this as a follow-up: *"Production orchestrator entry points still use the parameterless Load overload; migrating them to `Load(path, new SqlIdentifierValidator())` would close T-37-03-06 fully."* This plan IS that follow-up â€” but implemented by changing the default, not by touching 22 call sites (smaller diff, same coverage).

**Why not the per-predicate guard in `SqlTableProvider`?** That was the alternative fix path proposed in VERIFICATION.md. We rejected it because:
1. Validation at config-load is the single cheapest choke point â€” one place, early failure, clear error.
2. `SqlTableProvider.Serialize/Deserialize` would need the validator injected via DI, duplicating work already done.
3. Config-load validation also catches `ExcludeFields`, `IncludeFields`, `XmlColumns`, `ResolveLinksInColumns`, and `Where`-clause identifiers â€” the provider-level guard would only check `Table`.

**Test-suite compatibility â€” assembly-wide via a shared fixture.** Many tests across MULTIPLE test classes call the 1-arg `ConfigLoader.Load(path)` with valid SqlTable predicates. Because xUnit's AsyncLocal does not propagate across test classes (each class ctor runs in a fresh async context), a single ConfigLoaderTests-only installer is insufficient. The fix uses an xUnit collection fixture pattern: a shared `ConfigLoaderValidatorFixtureBase` abstract class installs the permissive AsyncLocal override in its ctor and clears it in Dispose; every affected test class inherits it, so every test method's async context gets the override before the test runs. This is the standard xUnit pattern for per-async-flow state like AsyncLocal.

**Out of scope:**
- Touching any of the 22 call sites of `ConfigLoader.Load(path)` â€” the fix happens in the overload, not the callers
- Adding belt-and-braces per-invocation validation in `SqlTableProvider.Serialize/Deserialize` (rejected above)
- Wiring `ProviderRegistry.CreateDefault` to inject a `SqlIdentifierValidator` â€” not needed once the 1-arg overload self-validates
- Any changes to `SqlIdentifierValidator`, `SqlWhereClauseValidator`, `ValidateIdentifiers`, `RuntimeExcludes`, or the admin UI surface (all delivered in 37-03 and verified correct)
- Addressing any other REVIEW.md finding (CR-02 symlinks, WR-01 thread-safety, WR-04..WR-11, IN-01..IN-09) â€” those are separate concerns not blocking SC-3
</objective>
