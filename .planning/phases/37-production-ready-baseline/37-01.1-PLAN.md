<objective>
Close the per-mode gap left by 37-01: wire `DeploymentMode` through the Item Type and XML Type admin UI (models, queries, commands, edit screens, tree) so that `Deploy.ExcludeFieldsByItemType` / `Deploy.ExcludeXmlElementsByType` and `Seed.ExcludeFieldsByItemType` / `Seed.ExcludeXmlElementsByType` are both independently editable through the admin UI. Remove the legacy `SerializerConfiguration.ExcludeFieldsByItemType` / `ExcludeXmlElementsByType` aliases that currently forward every read/write to `Deploy` only.

This is a gap-closure for D-02 (admin tree reflects the Deploy/Seed split). 37-01 split the tree for *predicates* but left the Item Types and XML Types subtrees at the shared root, where they silently read and write only Deploy's dictionaries — Seed's per-mode dictionaries exist in the data model (ModeConfig) and file format (ConfigWriter) but cannot be edited via the UI.
</objective>

<context>
**Current state (post-37-01):**
- `ModeConfig.ExcludeFieldsByItemType` and `ModeConfig.ExcludeXmlElementsByType` are per-mode in the data model.
- `ConfigLoader` / `ConfigWriter` correctly serialize these per-mode under `Deploy` and `Seed` sections.
- `PredicateEditModel`, `PredicateListQuery`, `PredicateByIndexQuery`, `SavePredicateCommand`, `DeletePredicateCommand` all carry `Mode: DeploymentMode` — done by 37-01.
- BUT `ItemTypeEditModel`, `XmlTypeEditModel`, `ItemTypeListQuery`, `ItemTypeBySystemNameQuery`, `XmlTypeListQuery`, `XmlTypeByNameQuery`, `SaveItemTypeCommand`, `SaveXmlTypeCommand`, `ScanXmlTypesCommand` — **none carry a Mode field**.
- These commands/queries all read/write through `SerializerConfiguration.ExcludeFieldsByItemType` / `ExcludeXmlElementsByType`, which are legacy alias properties forwarding to `Deploy` only.
- `SerializerSettingsNodeProvider` renders `ItemTypesNodeId` and the XML Types section at the shared root (line 84, line 100) — they are NOT children of the Deploy or Seed mode nodes.

**What good looks like:**
- Under the Deploy mode node in the tree: "Predicates", "Item Types", "XML Types" (three children).
- Under the Seed mode node in the tree: "Predicates", "Item Types", "XML Types" (three children).
- Opening any leaf under Deploy/Item Types edits `Deploy.ExcludeFieldsByItemType`.
- Opening any leaf under Seed/Item Types edits `Seed.ExcludeFieldsByItemType`.
- The legacy aliases are gone. All code reads/writes through `config.Deploy` or `config.Seed` explicitly.
</context>

<threat_model>
Security enforcement: ASVS L1. This plan does not add a new attack surface — all admin UI paths already require admin authentication via the DW admin pipeline. The only tampering concern is the `Mode` parameter on Item Type / XML Type commands:

| ID | Threat | Mitigation |
|----|--------|------------|
| T-37-01.1-01 | Tampering: attacker submits an invalid `Mode` string on SaveItemTypeCommand/SaveXmlTypeCommand to trigger dictionary misrouting | `Mode` is typed `DeploymentMode` (enum) — model binding rejects anything that is not `Deploy` or `Seed` at the framework boundary; no manual parsing in command handler |
| T-37-01.1-02 | Information disclosure: reading a different mode's excludes by guessing leaf URLs | Leaf URLs already require admin auth; both Deploy and Seed dictionaries are visible to the same admin user — no privilege tier between modes |
| T-37-01.1-03 | Data integrity: save-race between two admins editing the same (Mode, SystemName) tuple | Pre-existing concern inherited from 37-01 predicate save path — last-writer-wins is the current contract; documented as a known limitation (not in scope to fix here) |

Block on: none — no high-severity threats introduced.
</threat_model>
