<objective>
Broaden the schema/type resilience delivered in commit `f0bfbba` (Area-only) to every
raw-SQL write path in the codebase, make serialize runs self-cleaning, and build the
registry of known-runtime-only columns so users don't have to discover them by leaking.

Commit `f0bfbba` proved the approach on the Area table. This plan extends it
systematically to Page/Paragraph/GridRow extension writes and every ItemType_*
target, and adds the orthogonal pieces that came out of the same test session:
stale-output cleanup and the runtime-columns registry.

Does NOT change the DW-API-driven content paths (SavePage, ItemService) — those
already tolerate schema drift at the library level.
</objective>

### Task 1 — Extract schema-tolerant write helper

Commit `f0bfbba` introduced `GetTargetAreaColumns()` / `CoerceForColumn()` local to the
Area write path. Generalize to `TargetSchemaCache` with per-table keys:

- `TargetSchemaCache.GetColumns(tableName) -> HashSet<string>`
- `TargetSchemaCache.GetColumnTypes(tableName) -> IReadOnlyDictionary<string, string>`
- `TargetSchemaCache.Coerce(tableName, columnName, value) -> object`

One INFORMATION_SCHEMA query per table per deserialize run. Inject into
`ContentDeserializer` and `SqlTableWriter`.

### Task 2 — Apply to all raw-SQL write paths in ContentDeserializer

Audit every `Database.ExecuteNonQuery(cb)` and `Database.ExecuteScalar(cb)` in
`ContentDeserializer.cs` whose CommandBuilder builds an UPDATE/INSERT:

- `WriteAreaProperties` / `CreateAreaFromProperties` — already done (f0bfbba)
- Any extension-property writes on Page (check for raw UPDATE [Page] patterns)
- Paragraph extension-property writes
- GridRow extension-property writes
- ItemType_* field writes (SaveItemFields)

Each call site: filter + coerce before composing the SQL.

### Task 3 — Apply to SqlTableWriter

`SqlTableWriter.MergeRow` / `InsertRow`:

- Skip rows' columns not present on target (warn once per column)
- Coerce types before the parameterized write

### Task 4 — FileSystemStore atomic serialize

`FileSystemStore.WriteTree` today writes in-place into SerializeRoot without cleanup.
Change to:

1. Write every file to `SerializeRoot.new/` during the run
2. On success: remove `SerializeRoot/`, rename `SerializeRoot.new/` → `SerializeRoot/`
3. On failure: leave `SerializeRoot/` untouched; `SerializeRoot.new/` remains for
   debugging (cleared on next run)

Cross-platform note: Windows doesn't allow `rename` to replace an existing dir
atomically. Fallback: move `SerializeRoot/` → `SerializeRoot.old/`, rename
`.new` → `SerializeRoot/`, delete `.old` on success. On crash, startup code can
recover from `.old/`.

### Task 5 — RuntimeColumnsRegistry

Static class, initialized with a map:

```csharp
{
  "UrlPath":   ["UrlPathVisitsCount"],
  "EcomShops": ["ShopIndexName", "ShopIndexRepository", "ShopCreated"],
  "EcomOrders":/* future */,
  // ...
}
```

Called from `SqlTableReader` at serialize time: columns are excluded unless the
user has them explicitly in `excludeFields` OR has an `includeFields` override
(new, optional, opts back IN when someone knows what they're doing).

Document the list in README and in the reasoning template.

### Task 6 — FK-integrity pre-check (F-18)

In `SqlTableWriter` before MERGE/INSERT:

- For each FK column on the target, verify the parent row exists in source YAML OR
  target DB (use TableMetadata.ForeignKeys already populated)
- If orphan: log `ERROR: row {key} has FK to {parentTable}.{parentKey}={val} which
  does not exist in source or target — skipping`
- `result.Failed++` for each; orchestrator rolls up

Today the serializer silently disables FKs during writes, then can't re-enable
them when orphans exist. After this change: refuse to write the orphan,
re-enable FKs cleanly.
