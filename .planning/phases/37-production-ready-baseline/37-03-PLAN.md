<objective>
Stop the baseline from being a vehicle for credential leakage and identifier-based SQL
injection.

**Credentials**: Today a customer who fills in payment gateway secrets on Swift 2.2 and
serializes with the stock config leaks those secrets into git. `PaymentMerchantNum`,
`PaymentGatewayMD5Key`, CarrierAccountNumber fields in XML params — none of these
should ever be in a version-controlled baseline. Ship a curated default-exclude list
and require explicit opt-in to include them.

**Identifiers**: Today any string passed as a table name, column name, or where-clause
column is spliced directly into SQL (SEED-002 background). Plan 37-01 ships the
validator for the where-clause case; this plan broadens it to every identifier in the
config model.

**Out of scope**: encrypting what's in the baseline (that's Key Vault's job), or
scrubbing historical baselines that already contain credentials (that's a
runbook — document but don't automate).
</objective>

### Task 1 — CredentialColumnsRegistry

Static map, maintained in one place, readable as a policy document:

```csharp
public static readonly IReadOnlyDictionary<string, IReadOnlyList<string>> Defaults = new Dictionary<string, IReadOnlyList<string>>
{
    // Payment gateways: MD5 keys, merchant numbers, API tokens
    ["EcomPayments"] = new[] { "PaymentMerchantNum", "PaymentGatewayMD5Key", "PaymentGatewayId" },
    // XML-embedded: PaymentCheckoutParameters / PaymentGatewayParameters can contain secrets;
    // excluded at the element level via CredentialXmlElementsRegistry (not this registry)

    // Shipping: carrier account numbers and API tokens live in XML
    ["EcomShippings"] = new[] { /* row-level credentials if any */ },

    // Shop: env-specific search index bindings (crosses into runtime-only territory but
    // kept here since they're also env-configurable secrets for some setups)
    ["EcomShops"] = new[] { "ShopIndexRepository", "ShopIndexName" },

    // Document the rationale per entry in XML comments above each array.
};
```

Also: `CredentialXmlElementsRegistry` for per-item-type XML element exclusions
(e.g. `<CarrierToken>`, `<ApiKey>`, `<ClientSecret>`, `<MerchantId>` inside
`PaymentGatewayParameters`).

### Task 2 — Reader integration

`SqlTableReader.Read(predicate)`:

- `excludedColumns = predicate.ExcludeFields ∪ CredentialColumnsRegistry.Defaults[table]`
- If predicate.IncludeFields contains a default-excluded column: remove from
  excludedColumns, log `WARNING: predicate '{name}' opts back into serializing
  known-credential column [{col}] — verify target env has this value protected`
- Emit the SELECT with the effective column list

### Task 3 — Writer integration

`SqlTableWriter.Write`:

- If a row's YAML is missing a column that's in the default-excluded list AND the
  target already has a value, PRESERVE the target value (don't null it). This is
  the anti-source-wins rule: source-wins-null-out must not nuke a populated
  credential just because the baseline omits it.

Today `SavePage`-style paths in ContentDeserializer already have "source-wins
null-out" per PROJECT.md decision. Add an explicit exception: if the column is
in `CredentialColumnsRegistry.Defaults`, treat source-absent as "leave target
alone" not "null out."

### Task 4 — SqlIdentifierValidator (extend from 37-01)

37-01 shipped validation of where-clause columns. Broaden to:

- `ValidateAll(SerializerConfiguration config) -> List<string> errors`
- For each predicate: validate Table, NameColumn, every entry in ExcludeFields,
  IncludeFields, XmlColumns, ExcludeXmlElements column-ish entries against
  INFORMATION_SCHEMA
- For each top-level ExcludeXmlElementsByType key: validate as a C# type name
  (not a SQL identifier)

Call from `SerializerConfiguration.Load` AND `SavePredicateCommand` AND at
serialize/deserialize command entry so bad configs fail pre-query.

### Task 5 — Admin UI hygiene

`SavePredicateCommand`:

- On save, run the validator; block save with inline errors if any identifier
  fails. Use the same error structure other admin screens use (Edit screen
  validation infrastructure already exists; reuse).
