    **Create `LogViewerModel.cs`:**
    A model class that loads log file data. Properties:
    - `AvailableLogFiles` (List of file name strings, most recent first)
    - `SelectedFileName` (string, currently viewed file name)
    - `Summary` (LogFileSummary? parsed from selected file's JSON header)
    - `RawLogText` (string, the full log text after the summary header)
    - `HasLogs` (bool, whether any log files exist)

    In the model's data retrieval (constructor or query handler), resolve the log directory from config:
    ```csharp
    var configPath = ConfigPathResolver.FindConfigFile();
    if (configPath != null)
    {
        var config = ConfigLoader.Load(configPath);
        var filesRoot = Path.GetDirectoryName(configPath)!;
        var systemDir = Path.Combine(filesRoot, "System");
        var paths = config.EnsureDirectories(systemDir);
        var logFiles = LogFileWriter.GetLogFiles(paths.Log);
        // Populate AvailableLogFiles, load selected file, parse summary
    }
    ```

    Follow the pattern from `SerializerSettingsModel` / `SerializerSettingsQuery` for how the model gets populated via the query. Look at how `SyncSettingsQuery` (now `SerializerSettingsQuery`) implements `GetModel()` to understand the data loading pattern.

    **Create `LogViewerScreen.cs`:**
    Extend `EditScreenBase<LogViewerModel>` (read-only, no save command needed -- override `GetSaveCommand()` to return null or omit).

    In `BuildEditScreen()`, create sections:

    1. **Log File Selection** section:
       - A Select editor bound to `SelectedFileName` with options from `AvailableLogFiles`
       - When the selection changes, the screen should reload with the new file (the Select editor's change action should navigate back to LogViewerScreen with the selected file in the query)

    2. **Summary** section (only if `Summary` is not null):
       - Display `Summary.Operation` and `Summary.Timestamp`
       - Display `Summary.DryRun` status
       - Display a summary table: Total Created / Updated / Skipped / Failed
       - Per-predicate breakdown as a read-only list showing Name, Table, Created, Updated, Skipped, Failed

    3. **Advice** section (only if `Summary?.Advice` has items):
       - Display each advice string as a read-only text field or info message

    4. **Log Output** section:
       - Display `RawLogText` in a read-only multiline text editor (use `TextArea` or equivalent DW editor with ReadOnly=true)

    Use DW's `EditorFor()` pattern for simple fields. For the per-predicate table, use a `ListEditor` or repeated `EditorGroup` entries -- follow whatever pattern the existing `PredicateListScreen` uses for displaying lists.

    `GetScreenName()` should return `"Log Viewer"`.

    If no save is needed, check how DW handles read-only EditScreenBase. The simplest approach: still provide a `GetSaveCommand()` that does nothing, or check if there's a `ReadOnlyEditScreen` base class. If not, just return a no-op command. Look at the DW CoreUI patterns in the read_first files.
